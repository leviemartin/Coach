Architecture Approaches: Unified Coaching Flow

Three ways to restructure the data flow. All solve the same core problems — they differ in how aggressive the restructuring is.

A

Daily Log as Hub — Incremental Refactor

Keep existing pages but rewire the data flow. Daily log becomes the authoritative data source. Checkin pulls from daily logs instead of asking you to re-enter.

Daily Log (enhanced fields) ──writes──▸ SQLite daily_logs
Session Tracker ──on complete──▸ Daily Log (auto-populates)
Garmin Sync ──enriches──▸ garmin_coach_data.json
Checkin ──reads all──▸ daily_logs + sessions + Garmin
Triage Agent ──asks──▸ You answer
Coaches Run ──produce──▸ Draft Plan
Head Coach Chat ──refine──▸ Committed Plan
Plan ──distributes──▸ Daily Log (next week's sessions)

Pros

  • Least disruption — existing pages stay
  • Checkin form simplifies (fewer manual fields)
  • Can ship incrementally
  • Session tracker unchanged

Cons

  • Still 4 separate pages — navigation overhead
  • Plan page becomes redundant but lingers
  • Triage + dialogue bolted onto existing checkin flow
  • Data model patched, not redesigned
B

Daily Log as Hub — Full Consolidation

Daily log becomes THE page you use every day. Plan view is embedded. Session starts from daily log. Checkin is a Sunday mode of the daily log, not a separate page. Two primary pages: Daily Log (daily + weekly + checkin) and Session Tracker (in-gym).

DAILY LOG PAGE (your daily home)
├─ Day View: today's session + checklist + notes (Mon-Sun)
├─ Week Overview: expandable plan grid + compliance (always accessible)
├─ Sunday Checkin Mode: review + subjective inputs + triage Q&A (20:00+)
├─ Coach Synthesis: specialist outputs + Head Coach draft (after submit)
└─ Head Coach Chat: dialogue + plan refinement + commit (until satisfied)

SESSION TRACKER (in-gym, launched from daily log)
└─ On complete ──▸ writes summary back to daily log

Pros

  • One page to rule them all — minimal navigation
  • Checkin is a natural weekly view, not a separate flow
  • Data flows in one direction: daily → weekly → coaches
  • Plan embedded = no redundant page
  • Feels cohesive — everything about your training life in one place

Cons

  • Larger refactor — more work to ship
  • Page complexity grows (day view + week view + checkin mode)
  • Risk of overloading one page with too many responsibilities
  • Need clear mode transitions (daily → weekly → checkin)
C

Phased Hub — Best of Both

Daily log is the hub (like B), but the checkin stays its own page/route — accessed FROM the daily log when it's Sunday 20:00+. This keeps the daily log focused on daily tasks and the checkin focused on the weekly coaching conversation. Two daily pages, one weekly page.

DAILY LOG /log
├─ Day view with enhanced fields + session summary
├─ Expandable week overview (plan grid)
├─ Start session button → Session Tracker /session
└─ Sunday 20:00+: "Ready to check in" → Checkin /checkin

CHECKIN /checkin (redesigned)
├─ Step 1: Auto-assembled weekly review (from daily logs + sessions + Garmin)
├─ Step 2: Subjective inputs only (readiness, satisfaction, free text)
├─ Step 3: Triage agent Q&A (3-5 questions)
├─ Step 4: Coach synthesis (streaming specialists + Head Coach)
└─ Step 5: Head Coach dialogue → commit plan

SESSION TRACKER /session (unchanged)
└─ On complete → writes to daily log

Pros

  • Clear separation: daily tasks vs. weekly coaching conversation
  • Checkin page is purpose-built for the dialogue flow
  • Daily log stays clean and focused
  • Can ship daily log changes first, then checkin redesign
  • Each page has one job, done well

Cons

  • Still two pages to maintain (log + checkin)
  • Navigation between them needs to feel seamless
  • Slightly more surface area than full consolidation

My recommendation: C (Phased Hub). It gives you the unified data flow of B without overloading a single page. The daily log is your daily companion — lightweight, fast, focused. The checkin is your weekly coaching session — rich, conversational, purposeful. The session tracker stays what it is — an in-gym tool. Three pages, three clear jobs, one data flow.
